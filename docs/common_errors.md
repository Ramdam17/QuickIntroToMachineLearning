# Common errors & intuition traps

A curated, growing list of the narrow, recurring confusions in this course, so an AI tutor diagnoses
fast and correctly instead of reasoning from scratch. Add a row whenever a learner hits a new one;
point to the notebook that addresses it.

## Methodology traps (any method)

| Symptom / confusion | What's really going on | Where it's addressed |
|---|---|---|
| "My accuracy is 100% on the data I trained on." | You measured memorization, not generalization. Evaluate on held-out data. | `00_GettingStarted` |
| Scaling/feature selection done on the whole dataset before the split | Leakage — test info bleeds into training. Fit preprocessing on train only. | `00_GettingStarted`, `01_KNN/02` |
| "I changed the seed and the result changed — which is right?" | Single-seed results are noisy; fix and document the seed, and judge over CV folds. | per method, notebook 4 |
| "I picked the k (or any hyperparameter) that scored best on the test set." | That tunes the model to the test set, so the reported score is optimistic. Choose hyperparameters by cross-validation on the training data; touch the test set once, at the end. | `00_GettingStarted` (NB 10), `01_KNN/03` |
| "I'll report the best cross-validation score from my grid/sweep." | Optimistic too — you kept the luckiest setting (winner's curse). Estimate with nested CV (inner tunes, outer estimates), or a final sealed-test score. | `01_KNN/06` |
| Comparing models on different splits/preprocessing | Not a fair comparison. Same splits, same pipeline. | per method, notebook 5 |
| Accuracy looks great on imbalanced data | A constant predictor can score high. Use a metric that respects the minority class. | `02_NaiveBayes/05` |

## Method-specific (seed entries — extend during authoring)

| Method | Trap | Note |
|---|---|---|
| KNN | "Distances don't need scaling." | A large-range feature dominates the distance; normalize. |
| KNN | "The single nearest neighbour (k=1) is the most reliable guide." | One nearest point can be a noise stray; the majority vote over k≥3 neighbours is more robust near class overlap. The vote: `01_KNN/01`; choosing k: `01_KNN/03`. |
| KNN | "Odd k avoids ties, so I'm safe." | True only for **two** classes. With 3+ classes an odd k can still tie (e.g. 1-1-1 at k=3); sklearn then breaks it by the lowest class label. | `01_KNN/04` |
| KNN | "More features can only help." | For distance-based k-NN, irrelevant/noise features make all points look equally far (the curse of dimensionality; near/far distance ratio → 1), so accuracy *falls*. Select/reduce features. | `01_KNN/05` |
| KNN | "A fancier distance (Mahalanobis, fractional p) will rescue my k-NN." | The metric matters mainly in high dimensions; in low-d, well-scaled data it is a wash. Fix scale (NB 2) and cut dimensions (NB 5) first. | `01_KNN/06` |
| Naive Bayes | "Features must really be independent." | The assumption is *conditional* independence, and the method is often useful even when it's violated. |
| Naive Bayes | "A likelihood/posterior of exactly 0 or 1 is a hard fact." | A category never seen in a class gives likelihood 0, which zeroes the whole product (the zero-frequency problem) — it means *unobserved*, not impossible. Smoothing (add a little to every count) cures the overconfidence. Surfaced in `02_NaiveBayes/01`; fixed in `02_NaiveBayes/04`. |
| Naive Bayes | "The features are correlated overall (r≈0.87), so naive Bayes must fail." | The naive assumption is about *within-class* (conditional) correlation; the overall number is mostly the gap *between* classes. Measure the within-class value (here 0.33/0.66) — and even when it's nonzero, the *decision* often survives (the probabilities suffer more). `02_NaiveBayes/02`. |
| Naive Bayes | "The product of the per-feature likelihoods is 0 for every class." | With many features the product underflows to 0.0 in floating point — it's *unrepresentable*, not actually zero. Work in **log-space**: sum log-likelihoods instead of multiplying; the argmax is unchanged. `02_NaiveBayes/03`. |
| Naive Bayes / densities | "A probability density above 1 means something is wrong." | For a continuous feature it's the **area** under the density, not its height, that is a probability; a narrow distribution can have a peak above 1 while still integrating to 1. `02_NaiveBayes/03`. |
| Naive Bayes | "Turn `var_smoothing` (or `alpha`) up to be safe." | Smoothing is a floor, not a free lunch: too much inflates every variance (or flattens every count) until the classes stop differing and accuracy collapses toward the majority baseline. It's a safety net to tune, not a knob to max out. `02_NaiveBayes/04`. |
| Naive Bayes | "It reported 99%, so it's almost certainly right." | NB is **over-confident**: treating correlated features as independent double-counts evidence and piles probabilities at 0/1. Trust its *ranking*, not the raw number; recalibrate (`CalibratedClassifierCV`) if you need a probability. `02_NaiveBayes/05`. |
| Logistic Regression | "It's regression." | It's a classifier; the output is a probability via the sigmoid. |
| Logistic Regression | "The linear score w·x+b is the probability." | The score is unbounded (any real number) — it is the **log-odds**, not the probability; pass it through the sigmoid σ(z)=1/(1+e⁻ᶻ) to land in (0,1). (The log in the logit is the **natural** log, base e — the inverse of σ's exponential.) `03_LogisticRegression/01`. |
| Logistic Regression | "A larger weight means a more important feature." | True only on a shared scale: on raw features a large weight can merely reflect a feature measured in small units (the scale trap). Standardize first, then compare magnitudes — each weight is a change in log-odds per standardized unit. `03_LogisticRegression/02`. |
| Logistic Regression | "Train it by minimizing squared error, like a regression." | Through the sigmoid, squared error is **non-convex** and flattens into plateaus where the gradient vanishes and a descent stalls; its penalty for a confident-and-wrong prediction is capped at 1. Train with **log-loss** (cross-entropy = negative log-likelihood): convex (one minimum) and unbounded on confident mistakes. `03_LogisticRegression/03`. |
| Logistic Regression | "A bigger learning rate trains faster." | Up to a point: too small and gradient descent crawls; too large and each step overshoots the minimum, so the loss climbs instead of settling. The usable range shrinks to a knife-edge on un-standardized features — standardize, then tune the rate. `03_LogisticRegression/04`. |
| Logistic Regression | "Larger `C` means more regularization." | `C` is *inverse* strength: large `C` = **weak** penalty (bigger weights, more complex); small `C` = strong penalty. And the penalty *type* is set by `l1_ratio` (0 = L2, 1 = L1), not the deprecated `penalty=`. `03_LogisticRegression/05`. |
| Logistic Regression | "0.5 is the right decision threshold." | 0.5 is only the default. When one mistake costs more than the other (a missed cancer vs a false alarm), move the threshold to trade them — lower it to catch more, at more false alarms — and trust that choice only because the probability is calibrated. `03_LogisticRegression/06`. |
| Decision Tree | "I should standardize the features first (as for k-NN / logistic regression)." | A split is a threshold test ("is x ≤ t?"), so any monotone rescaling only moves the threshold — the partition, and the whole tree, is unchanged. Trees are **scale-invariant**; the Chapter-01 scale trap does not apply. `04_DecisionTree/01`. |
| Decision Tree | "I can see the best feature to split on from the scatter." | Eyeballing is a hint, not a choice. Score each candidate split by its **impurity decrease** (Gini/entropy) and let the number pick the feature and threshold. `04_DecisionTree/01`. |
| Decision Tree | "A trained tree is the best possible set of questions." | Growth is **greedy** — each split is only the locally best one, never looking ahead; the globally optimal tree is NP-hard (Hyafil & Rivest 1976). The grown tree is good and fast to build, not provably optimal. `04_DecisionTree/02`. |
| Decision Tree | "Deeper is better." | Depth is the complexity dial: too shallow underfits, too deep memorizes (training error → 0 while test error climbs). Tune depth (and pruning) by cross-validation, never on the test set. `04_DecisionTree/03`. |
| Decision Tree | "The thin horizontal/vertical bands in a deep tree's decision boundary look like a plotting bug." | They are **real axis-aligned regions**: a deep tree carves a narrow box (a "fence") around individual noisy points — that *is* overfitting. Pruning removes them. `04_DecisionTree/03`. |
| Decision Tree | "Pruning only trades accuracy away for a simpler tree." | Pruning a tree that has memorized the training set can *raise* held-out accuracy by dropping noise-fitting splits; choose the strength (`ccp_alpha` or `max_depth`) by cross-validation. `04_DecisionTree/03`. |
| Decision Tree | "This tree scored 0.90 on the test set, so that is how good a decision tree is here." | A single tree is **high-variance**: refit it on a resample and the boundary — and the score — can swing (e.g. 0.900 vs 0.833 on the *same* test set, both trees 17 leaves). Trust a cross-validated estimate, and cut the variance by averaging many trees (random forests, ch 06). `04_DecisionTree/04`. |
| Decision Tree | "The built-in feature importance named feature X, so X is the important one." | Gini (impurity) importance is biased and can *disagree* with permutation importance — on correlated features it over-credits whichever was split on first (e.g. Gini → `mean concave points`, permutation → `worst area`). Cross-check with permutation importance (shuffle a feature, measure the held-out accuracy drop). `04_DecisionTree/05`. |
| SVM | "I don't need to standardize the features." | An SVM measures distances (the RBF kernel is a function of ‖x−x′‖, and `gamma='scale'` depends on feature variance), so a large-range feature dominates — the k-NN scale trap (ch 01) returns. Standardize, fit-on-train-only. `05_SVM/05`. |
| SVM | "The two closest opposite-class points are always the support vectors (and the boundary is their perpendicular bisector)." | True only when a single pair pins the street. In general the max-margin boundary runs between the classes' **convex hulls**, so a hull *edge* can face the other class and **three or more** points become support vectors. The closest-pair recipe is a clean special case, not the rule. `05_SVM/01`. |
| SVM | "Every support vector is a point the model got wrong, or one that pays a penalty." | A support vector is any point with functional margin `m = y(w·x+b) ≤ 1` — on **or** inside the street. Those resting *on* the edge (`m = 1`) pay **zero** slack and are classified correctly; only those inside it or on the wrong side pay the hinge. So `#support-vectors ≥ #penalty-payers ≥ #misclassified` (e.g. 17 ≥ 15 ≥ 2). `05_SVM/02`. |
| SVM | "A bigger `C` is better — turn it up to be safe." | `C` is a bias–variance dial, not a quality knob: large `C` narrows the margin (toward the hard margin, more variance), small `C` widens it (more bias, more support vectors). On near-separable data it changes the *geometry* more than the accuracy. Tune it by cross-validation. `05_SVM/02` (and `/04`). |
| SVM | "A kernel (or a high-enough polynomial degree) fits any curved boundary automatically." | No — the kernel and its knobs must **match the data's geometry**. On concentric rings the RBF and a degree-**2** polynomial succeed, but the *default* degree-3 polynomial nearly fails: with `coef0=0` it builds only odd-power terms, so it cannot form the radial `x₁²+x₂²`. Choose and tune the kernel (degree, `gamma`); never assume one fits. `05_SVM/03`. |
| SVM | "A bigger `gamma` is better." | `gamma` is the RBF's reach and a bias–variance dial, not a quality knob: too small underfits (broad reach, near-linear boundary), too large overfits (tiny bubbles around each point). Read the `C × gamma` map and pick the cell by cross-validation. `05_SVM/04`. |
| SVM | "`SVC`'s `decision_function` is a probability (or `probability=True` gives a calibrated one)." | `decision_function` is a signed-distance **score** (its sign is the class, its magnitude the confidence), not a probability. For a probability, wrap the model in `CalibratedClassifierCV` (Platt / sigmoid); `SVC(probability=True)` is **deprecated** in 1.9 — use `CalibratedClassifierCV(SVC(), ensemble=False)`. `05_SVM/04`. |
| SVM (any scored model) | "Lowering the decision threshold will catch the positives I'm missing." | Only the *borderline* misses near the boundary. A point the model is **confidently wrong** about sits far on the wrong side (calibrated probability well below any sensible cut), so no threshold recovers it — lowering the cut only adds false alarms. Look at *where* the misses actually sit before reaching for the threshold. `05_SVM/05`. |
| Random Forest | "More trees will keep making the forest better." | Bagging averages many high-variance trees to cut **variance**, with **diminishing returns** (the run-to-run spread shrinks roughly like σ²/B, then plateaus past a few dozen trees); it never reduces **bias**, so a forest of biased trees stays biased and extra trees only cost compute. `06_RandomForest/01`. |
| Random Forest | "Bagging is enough — just add more trees." | Bagged trees are **correlated** (ρ ≈ 0.82 here): the variance of their average is ρσ² + (1−ρ)σ²/B, so more trees (B↑) only kill the second term and leave a **floor of ρσ²** no number of trees can pass. Lowering ρ — random **feature subsampling** at each split (`max_features`) — is the "random" in random forest, and the only way under the floor. `06_RandomForest/02`. |
| Random Forest | "Smaller `max_features` (more randomness) is always better — and it works on any data." | It decorrelates (lower ρ) but also weakens each tree, and it needs **many features** to pay: on a 2-feature problem `'sqrt'`→1 feature/split starves the trees (RF *below* plain bagging on `make_moons`); with 30 features it is near-free decorrelation. `'sqrt'` is the robust default; tune by CV. Also: a 1-point ensemble gain on a small test/CV set is within the seed band — trust the robust ρ drop, not a single accuracy delta. `06_RandomForest/02`. |
| Random Forest | "The OOB score is the forest's true accuracy — use it instead of a test set." | Out-of-bag is an honest *in-training* estimate (each point judged by the ~37 % of trees that never saw it), but **mildly optimistic** — here ~1–2 points above the sealed test — and **unreliable with few trees** (some points are never out-of-bag; sklearn warns). Use OOB as a cheap running estimate; report the sealed test (or nested CV) as the headline. `06_RandomForest/03`. |
| Random Forest | "Cap the trees' depth so the forest doesn't overfit." | A forest grows trees to **full depth on purpose** (`max_depth=None`, `min_samples_leaf=1`): each tree is low-bias / high-variance, and averaging cancels the variance a *lone* deep tree could not survive. Here a deep forest is ~4× steadier than a single deep tree (run-to-run std 0.004 vs 0.016) and more accurate; over-restricting the leaves only lowers CV. Deep trees are a feature, not a bug, once you average them. `06_RandomForest/04`. |
| Random Forest | "I must tune the forest hard to get good accuracy." | A random forest is **forgiving**: across `max_features` 1→30 the accuracy here is flat (0.95–0.96), and a full `GridSearchCV` beat the default by a fraction of a point (CV 0.955→0.957, test 0.942→0.947). Its appeal is *strong with almost no tuning*; `'sqrt'` is a robust default, and that dial's real job is decorrelation (ρ, NB 2), not accuracy. More trees never overfit but cost compute — pick "enough". `06_RandomForest/04`. |
| Random Forest | "The forest's top-importance feature is *the* important one (and importance means cause)." | A single tree pours ~0.74 of its importance onto whichever correlated feature it split on first; a forest **spreads** the credit across the correlated group (peak ~0.15) — fairer, but MDI is still **biased** toward continuous / high-cardinality features and **dilutes** across correlated / one-hot groups, and importance is **not causal**. Read at the *group* level; cross-check with permutation importance (NB 5). `06_RandomForest/04`. |
| Random Forest | "The model is 84 % accurate, so it is good." | Under **class imbalance**, accuracy (and weighted-F1) are dominated by the big classes — covtype's forest scores 0.844 accuracy yet catches only **28 %** of the rare Aspen class. **Macro-F1** (0.737, unweighted over classes) and **per-class recall** reveal what accuracy hides; the confusion matrix shows the rare class collapsing into its common look-alike. A forest does not fix imbalance — use `class_weight`/resampling. `06_RandomForest/05`. |
| Random Forest | "Each one-hot soil column has ~0 importance, so soil type does not matter." | A signal split across **40 one-hot dummies** is **diluted** per column (each ≈ 0) but the group **combined** is large (Soil ≈ 0.14 MDI / 0.11 permutation — the second-biggest signal after Elevation). Read importance at the **group level**; MDI's bias toward continuous/high-cardinality features compounds the dilution. `06_RandomForest/05`. |
| Random Forest | "A random forest is the best model — it won decisively." | It won on covtype **because the problem is non-linear**; on near-linear breast_cancer it **lost** to the SVM (RF ≈ 0.94 < 0.965). No method is best everywhere — the right tool tracks the problem's *shape*. A forest's claim is *strong with little effort*, and boosting often edges it with more tuning (ch 07–10). `06_RandomForest/05`. |
| AdaBoost | "Boosting is another ensemble vote, like bagging with more trees." | Opposite construction. Bagging trains trees **in parallel** on independent bootstraps and gives each an **equal** vote (cuts variance); boosting trains weak learners **in sequence**, each on data **reweighted** toward the previous rounds' mistakes, combined with **unequal** votes α (attacks bias *and* variance). More AdaBoost rounds is error-correction, not "more trees averaged". `07_AdaBoost/01`. |
| AdaBoost | "Round 2's weighted error went *up* vs round 1 — the boosting is going backwards." | By design: each round reweights the data to dwell on the prior mistakes (easy points down-weighted), so a later weak learner faces a *harder* reshaped problem and its weighted error ε can rise. A higher ε just earns a smaller vote α = ln((1−ε)/ε); the *ensemble's* training error still falls overall. `07_AdaBoost/01`. |
| AdaBoost | "The ensemble's training-error curve wiggles upward at some rounds — is that a bug?" | No. AdaBoost bounds the training-error *envelope*, not a monotone step-by-step decrease — adding one weak learner can momentarily raise the combined 0/1 error before later rounds pull it down. The trend falls overall (and *training* error falling is not yet generalization — that is NB 3). `07_AdaBoost/01`. |
| AdaBoost | "My textbook's α has a ½ (½ ln((1−ε)/ε)) but scikit-learn's doesn't — different algorithm?" | Same classifier, two conventions. ½ ln((1−ε)/ε) is the classic **vote** weight (the exponential-loss minimiser); SAMME (sklearn) uses ln((1−ε)/ε) = 2×. They agree because the classic *reweight* is the **margin form** exp(−β·y·h) = exp(−β)·exp(2β·𝟙[miss]) — the common exp(−β) cancels in renormalisation, leaving SAMME's exp(2β·𝟙[miss]); both build the **same stumps**, and the vote's *sign* ignores the global factor 2. (It is NOT "a factor on α cancels in renorm": α is in the exponent — reweighting misses by exp(β) instead of exp(2β) diverges.) `07_AdaBoost/02`. |
| AdaBoost | "α = ln((1−ε)/ε) is a definition to memorise." | It is *derived*, not decreed: α is the step that minimises AdaBoost's **exponential loss** each round (forward stagewise additive modelling) — the minimiser of L(α) = (1−ε)e^(−α) + ε e^(α) is ½ ln((1−ε)/ε). The exponential loss is a smooth **surrogate** for the 0/1 error; its blow-up on confidently-wrong points is the seed of AdaBoost's noise-sensitivity (NB 3/5). `07_AdaBoost/02`. |
| AdaBoost | "AdaBoost resists overfitting, so I can pile on rounds freely." | True only on **low-noise** data — a margin phenomenon (Schapire et al. 1998): training error can reach 0 while test error holds a flat band. It is **not immunity**. Under **label noise** the test error *rises* with rounds (the exponential loss chases the mislabeled points, carving islands around them): on a 25%-flip run the clean test drifts only ~0.017 from its minimum while the noisy test climbs ~0.083. Use **early stopping**, a smaller `learning_rate`, or a more robust loss (ch 08). `07_AdaBoost/03`. |
| AdaBoost | "`learning_rate` and `n_estimators` are independent knobs." | No — `learning_rate` ν scales **every** learner's contribution *and* its reweighting step (`estimator_weights_ = ν·ln((1−ε)/ε)`), so a smaller ν takes smaller steps and needs **more rounds** to reach the same fit. They trade off and are tuned **together** (small ν + many rounds is the usual sweet spot). `07_AdaBoost/03`. |
| AdaBoost | "A deeper (stronger) base learner will boost to a better model." | Boosting's bet is that **weak** learners combine into a strong one, so the base must stay weak. A deeper base **overfits faster**: on moons every base depth drives *training* accuracy to 1.000, but the *test* accuracy falls as the base deepens (stump 0.942 → depth-5 0.917) and the boundary grows visibly more contorted. This is the **mirror image of a random forest**, which wants *deep*, low-bias trees and averages their variance away; boosting wants *weak*, high-bias learners and reduces bias by adding many. Keep the default stump (or near it). `07_AdaBoost/04`. |
| AdaBoost (reading any `GridSearchCV` grid) | "I'll `reshape` `cv_results_['mean_test_score']` into my `(n_estimators, learning_rate)` matrix." | `GridSearchCV` walks the grid in **alphabetically-sorted key order** (`learning_rate` outer, `n_estimators` inner), not the order you wrote the dict — so a naive `reshape(len(n), len(lr))` silently **transposes** the matrix, mislabels the heatmap, and can contradict the printed `best_params_`. Rebuild the grid from the actual parameter values instead — e.g. `pd.DataFrame(grid.cv_results_).pivot_table(index="param_learning_rate", columns="param_n_estimators", values="mean_test_score")` — so the orientation cannot drift. `07_AdaBoost/04`. |
| AdaBoost | "AdaBoost shines on this dataset, so boosting is the best method." | It shines (spam test 0.949) but does not *win* — the random forest edged it (0.959), and on near-linear breast_cancer the **tuned SVM** beat the methods compared there. No model is universally best; the family that wins tracks the **shape** of the problem (spam's many weak token-counts suit a tree ensemble; a near-linear problem suits the margin method). AdaBoost's claim is *strong and competitive*, not *best everywhere*. `07_AdaBoost/05`. |
| AdaBoost | "Boosting is fragile to label noise, so a random forest is the safe choice under noise." | Dataset-dependent, **not a law**: on spam AdaBoost is the *more* robust of the two from 20% noise upward (at 40%: 0.82 vs RF 0.70), because its **weak stump** base cannot memorize individual mislabeled points while a random forest's **deep** trees can. The robust, dataset-independent statement is the *mechanism*: the exponential loss piles weight onto the mislabeled points (by hand, ~21% of points come to hold ~40% of the weight). Resistance is real but **not immunity** — under heavy noise more rounds hurt, so use early stopping or a gentler loss (ch 08). `07_AdaBoost/05`. |
| AdaBoost | "More rounds drove the training error to zero, so AdaBoost memorized the data." | Training error reaches exactly 0 only on clean, low-noise toys (the 2-D moons of NB 1–3). Real data carries **irreducible overlap**, so the training error floors *above* zero (spam ≈ 0.045) no matter how many rounds — and on clean data the **test** error still does not climb with more rounds (the resistance of NB 3). "Train error to zero" is a toy phenomenon, not a sign of memorization on real data. `07_AdaBoost/05`. |
| Gradient Boosting | "Gradient boosting is AdaBoost with trees — it reweights the points too." | Different mechanism, same goal (focus on the mistakes). AdaBoost **reweights points** toward past errors; gradient boosting **fits each new regression tree to the residuals** `y − F` of the current model and adds a shrunken slice. No sample weights — it chases the *leftovers* directly. `08_GradientBoosting/01`. |
| Gradient Boosting | "My by-hand gradient boosting will match `GradientBoostingRegressor` exactly, whatever the loss." | It matches to **machine precision for squared error** because a regression tree's leaf is already the residual **mean** — the value minimising squared error there, so no leaf re-estimation is needed. Other losses (and classification) need a **leaf correction** the library applies for you (the Newton step, NB 3). And only if you match init=mean, depth, ν, `subsample=1.0`, and seed. `08_GradientBoosting/01`. |
| Gradient Boosting | "I'll read its learning curve with `staged_score`, like AdaBoost." | `GradientBoostingRegressor`/`Classifier` expose `staged_predict` / `staged_decision_function` / `staged_predict_proba` but **no `staged_score`** (unlike `AdaBoostClassifier`). Compute the metric over `staged_predict` yourself, e.g. `[mean_squared_error(y, p) for p in est.staged_predict(X)]`. `08_GradientBoosting/01`. |
| Gradient Boosting | "Where is the gradient in gradient boosting? We only ever fit residuals." | For squared error, the residual `y − F` **is** the negative gradient of the loss `½(y−F)²`, so fitting it *is* one gradient-descent step — taken in **function space** (the variables are the n predictions, not weights). Change the loss and you fit a different negative gradient: absolute error → `sign(y − F)`, log-loss → `y − p` (NB 3). That choice of loss is the "gradient" in gradient boosting. `08_GradientBoosting/02`. |
| Gradient Boosting | "So gradient boosting does exact gradient descent." | Each tree is a **piecewise-constant approximation** of the negative gradient (one value per leaf — it cannot move two points in a leaf independently), so boosting takes many small *approximate* steps, not exact steepest descent. On a 2-D slice of function space the path can even ease slightly past the target before settling. `08_GradientBoosting/02`. |
| Gradient Boosting | "For classification, put the mean of `y − p` in each leaf, like a regression tree." | Wrong for log-loss. The optimal leaf value is the one-step **Newton update** `γ = Σ(y−p) / Σ p(1−p)` — it accounts for the loss curvature `p(1−p)` (the mean ignores it and under-steps, a *different*, slower model). `GradientBoostingClassifier` applies the Newton step silently; the by-hand mean-leaf does **not** match it. `08_GradientBoosting/03`. |
| Gradient Boosting | "`GradientBoostingClassifier(loss='exponential')` and `AdaBoostClassifier` are the same model." | They share the **exponential-loss objective** (identical test accuracy, ~98% prediction agreement here) but run **different optimizers** — GB takes functional-gradient steps with Newton leaves and shrinkage; AdaBoost reweights points and votes with α = ln((1−ε)/ε). Not bit-identical. AdaBoost is the exponential-loss *member* of the gradient-boosting family (objective-level, not predictor-level). `08_GradientBoosting/03`. |
| Gradient Boosting | "More trees is always safe — like a random forest, keep adding them." | **False for gradient boosting at a large ν.** Boosting is sequential — each tree fits the current residual and *adds capacity* — so once the residual is mostly noise, new trees fit the noise and the **test error turns up**: on friedman1 at ν=1, test R² peaks 0.864 @ 18 trees then sags to 0.813 @ 1000 while train MSE → 0. It is **ν-dependent** (at ν=0.1 the curve stays flat within budget). A random forest *averages independent trees*, so more trees only sharpen the average and never overfit (test R² ≈ 0.86 flat from B=50 to 1000). Same words, opposite mathematics — and the reason for early stopping (NB 5). `08_GradientBoosting/04`. |
| Gradient Boosting | "A larger learning rate trains faster, and ν and n_estimators are separate knobs." | ν is a **step-size brake** coupled to the number of trees: each tree enters as ν·tree, so a small ν moves a fraction of the way and needs **proportionally more trees** (to reach train MSE ≤ 2 on friedman1: 13 trees at ν=1, 48 at ν=0.1, 496 at ν=0.01). Yet the small step reaches a **better, safer** optimum (ν=0.1 test R² 0.930 vs ν=1's 0.864), because many gentle corrections lean on consistent signal instead of chasing noise. Shrinkage is a regularizer; **small ν + many trees** is the sweet spot — tune the two together. `08_GradientBoosting/04`. |
| Gradient Boosting | "Deeper trees fit better — turn up max_depth for accuracy." | Tree depth is an **interaction order**, not a quality knob. A stump (depth 1) splits on one feature, so an ensemble of stumps is **additive** and cannot represent an interaction like x₀·x₁ — on friedman1 it underfits (test R² 0.873, train only 0.905). Depth 2 captures the two-way interaction (0.931, the biggest single gain); depth 5 begins to **memorize** (train R² 0.998, test 0.923). Match depth to the problem's real interaction order and keep trees **shallow** (sklearn default 3) — the mirror image of a random forest's deliberately deep trees. `08_GradientBoosting/04`. |
| Gradient Boosting | "I have to pick `n_estimators` — I'll guess a big number or grid-search it." | Let the data choose: set a generous `n_estimators` and turn on **early stopping** (`validation_fraction` + `n_iter_no_change` + `tol`). It holds out a slice of the training data and stops when that score plateaus — on friedman1 it halted a **2000-tree** request at **142**, with test R² (0.930) a touch *above* the full model (0.927, by then gently overfit). Early stopping is **off by default** (`n_iter_no_change=None`); the stop point drifts with the seed (it tracks a held-out slice), the score barely does. `08_GradientBoosting/05`. |
| Gradient Boosting | "`subsample` < 1 only trades accuracy for speed." | It is a **regularizer** (stochastic gradient boosting, Friedman 2002): fitting each tree on a random fraction of the rows **decorrelates** the steps — the bagging idea (ch 06) inside a booster — and here `subsample` 0.5–0.75 **beat** full-sample on the test set (R² ≈ 0.936 vs 0.929) *and* cost less per tree. The companion `oob_improvement_` is a free running estimate but **noisier** than a held-out set; prefer validation-based early stopping for choosing the number of trees. `08_GradientBoosting/05`. |
| Gradient Boosting | "`feature_importances_` tells me which features truly matter." | It is **MDI** (impurity reduction) with the same biases as trees / forests (ch 04/06) — toward continuous / high-cardinality features — so cross-check with **permutation importance**. On friedman1 (known: 5 informative + 5 noise) both recover the signal (MDI over x₀–x₄ sums to 0.988) and zero the noise, and agree on the ranking — but importance is **use, not cause**, and reflects this dataset, not the world. `08_GradientBoosting/05`. |
| Gradient Boosting (any regressor) | "The target's $500k cap is just a bit of noise at the top." | It is **censored** data: every home at/above $500k was recorded as exactly 5.0, so the true values up there were *erased* — no model can learn them, and the error on expensive homes balloons (≈ $74k vs ≈ $25k on cheap homes, a 3× gap that begins *below* the cap). And a tree-sum is **not** bounded by the training maximum (a few predictions here exceed 5.0) — the limit is the missing labels, not the model's range. Surface the cap; report error by segment, not one MAE. `08_GradientBoosting/06`. |
| Gradient Boosting | "MDI and permutation importance basically agree." | They can **flatly disagree**. On California housing MDI says median income dominates (0.58) and location is minor (~0.10 each), while **permutation on held-out data says location dominates** — shuffling latitude or longitude collapses the model (≈ 3.2–3.4 R² drop). The reason is **interactions**: location's signal is spread over many deep, low-impurity splits (lat × lon × income) that MDI each scores as tiny, while permutation destroys the whole interaction at once. Trust the held-out permutation read here; read correlated features (lat/lon) as a pair. `08_GradientBoosting/06`. |
| any regressor | "One held-out R² (or MAE) tells me how good the model is." | A single number averages over very unequal cases. The California booster's $33k MAE hides a **3× spread** by price ($25k on cheap homes → $74k on the priciest), and a predicted-vs-actual plot reveals a wall at the $500k cap the metric cannot show. Read R² **and** MAE (in real units), broken down by segment, with a residual / predicted-vs-actual plot. `08_GradientBoosting/06`. |
| XGBoost | "Regression uses a mean leaf and classification a Newton leaf — two separate rules." | They are **one** second-order rule: the optimal leaf weight is `w* = −G/H`, with `G = Σ ∂L/∂F` and `H = Σ ∂²L/∂F²`. Squared error has constant curvature `h = 1`, so `−G/H` is the plain **mean residual** (ch 08 NB 1); log-loss has `h = p(1−p)`, so `−G/H` is the **Newton leaf** `Σ(y−p)/Σ p(1−p)` (ch 08 NB 3). The loss only changes the two sums — which is why XGBoost runs the same engine for *any* twice-differentiable loss. `09_XGBoost/01`. |
| XGBoost | "The gradient alone tells you how to update a leaf." | The gradient gives the **direction**; the **curvature** `h` gives the step length. Treating log-loss like squared error — taking the mean residual, i.e. assuming `h = 1` — under-steps; dividing by the true curvature `Σ p(1−p)` rescales the step onto the loss minimum. That is Newton's method, the move XGBoost makes at every leaf. One such step is the best *local* quadratic move, not the exact optimum (so boosting still iterates), and it requires `h > 0`. `09_XGBoost/01`. |
| XGBoost | "`reg_lambda` and `gamma` just trim the tree after it is built." | XGBoost prices complexity **inside the objective** `Ω = γ·T + ½λ·Σ w²`, which changes the math two ways. `λ` (L2) shrinks every leaf to `w* = −G/(H+λ)` — toward 0, like ridge / ch 03's `C`; `γ` is subtracted from each split's gain, so a split is kept only if it clears `γ` (pre-pruning). Complexity control is *built into* what is optimised, not bolted on — where ch 04/08 had only `max_depth` and the number of trees. `09_XGBoost/02`. |
| XGBoost | "Split gain is parent loss − children loss." | It is `score(before) − score(after)` of the **structure score** `−½ Σ G²/(H+λ) + γT` (C&G eq. 6), which is **negative** — so the intuitive "parent − children" flips the sign. Derive the gain from the score: `½[G_L²/(H_L+λ) + G_R²/(H_R+λ) − G²/(H+λ)] − γ` (eq. 7). A *positive* gain means the (negative) objective got *more* negative — the split helped. `09_XGBoost/02`. |
| XGBoost (reading `Gain` / tuning `gamma`) | "The gain XGBoost prints is the textbook `½[…]−γ`." | XGBoost reports the gain **without the ½** — exactly **twice** the eq. 7 value (measured: by-hand 9 → reported 18). It is internally consistent: the `γ` (`min_split_loss`) threshold is in those *same* no-½ units (measured: the split survives `γ ≤ 18`, prunes at `γ = 19`). The ½ scales gain and `γ` together, so the *decision* is identical — but tune `γ` in the units XGBoost prints as `Gain`, not the textbook's. Also: `Cover` in `trees_to_dataframe` is `Σ h`. `09_XGBoost/02`. |
| XGBoost | "XGBoost imputes the missing values for you." | It does **not** impute. At each split it sends every missing row down a single **learned default direction** — chosen by computing the split gain (NB 2) *both ways* (all-missing-left vs all-missing-right) and keeping the larger (Chen & Guestrin §3.4). The `NaN` stays a `NaN`; its routing is decided by which side raises the gain, so an informative missing pattern is **used**, not erased. `09_XGBoost/03`. |
| XGBoost / sklearn | "All the boosters handle a `NaN` the same way." | They do not. Plain `GradientBoosting*` **rejects** a NaN (`ValueError: Input X contains NaN` — you must impute first); `HistGradientBoosting*` (ch 08) and XGBoost **accept** it natively (XGBoost first, 2016). Native handling can beat impute-then-fit **when missingness is informative** — a column-mean impute throws away the fact that a value was missing. (CART's classic answer was *surrogate splits*, ESL §9.2.4 — a heavier mechanism.) `09_XGBoost/03`. |
| XGBoost | "XGBoost's defaults overfit (train hits 1.0), so the defaults are broken and tuning will fix the score." | The defaults **do** memorize — `max_depth=6` + `eta=0.3` drive train accuracy to exactly **1.0000** (2608 leaves on a 4200-row set). But the test is already near the data's **noise ceiling**, so honest tuning (GridSearchCV → one sealed test) buys little *accuracy* (**+0.003** here) and mostly a **more robust** model (smaller gap, gentler trees). And `max_depth` is an interaction-order dial, not a quality knob: test peaks at a moderate depth (4 here) and the default 6 is already **past** it — always sweep it. XGBoost's edge is speed / native NaN / regularization, not a guaranteed accuracy win. `09_XGBoost/04`. |
| XGBoost (tuning `reg_lambda`) | "XGBoost turns L2 on by default (`lambda=1`), so the default test score beats L2 off." | Not on noise-capped data: turning L2 **off** (`reg_lambda=0`) leaves the test a hair *higher* here (0.9456 vs 0.9444) — and train stays 1.0 either way, so `lambda=1` does **not** stop the memorization. `λ` shrinks every leaf weight `−G/(H+λ)`: its job is the **magnitude** of the predictions (calibration), not lifting accuracy. It only visibly closes the train/test gap when **large** (λ ≈ 100+); the everyday gap-closers are a smaller `max_depth`, a larger `gamma`, a larger `min_child_weight`. `09_XGBoost/04`. |
| XGBoost (`tree_method`) | "`hist` bins the features, so it must be less accurate than `exact`." | Measured: on 42,000×50 it fit **~5× faster** (≈4.2 s → 0.85 s) at **no accuracy cost** (0.9646 vs 0.9660 — `hist` a hair higher, within run-to-run noise). Binning to ≤`max_bin` (256) edges only shrinks the *set of candidate thresholds*; the gain formula (NB 2) is unchanged, the best split rarely falls strictly between bin edges, and coarse bins act as a mild regularizer. That is why `hist` is the 3.x default and how boosting scaled to millions of rows. `09_XGBoost/04`. |
| XGBoost (real data) | "Missingness is informative here, so XGBoost's native NaN handling will beat imputation." | **Measure it — do not assume.** On Adult the missing flag is strongly informative univariately (P(>50K) ≈ 0.09 missing vs 0.25 present), yet native-NaN, mode-imputed, and an explicit `MISSING` category all score the same PR-AUC (≈ 0.83; native−imputed Δ −0.001 that **flips sign across resplits** — split noise, not a deficit). The signal is **redundant**: the two informative-missing columns are nearly the same flag (occupation∩workclass 2799/2809) and what they carry is largely present elsewhere. Native handling's real value here is **convenience** (no preprocessing), not accuracy. `09_XGBoost/05`. |
| XGBoost vs the field | "XGBoost tops the comparison, so it is the stronger model." | Part of the edge is **preprocessing convenience**, not power. On Adult XGBoost/HistGBR (native NaN + categoricals) lead, but GradientBoosting is only a step behind largely because it needed ordinal-encode + impute — and at *matched* capacity on the *same* encoded+imputed inputs, XGBoost ≈ GradientBoosting (even a hair behind). No universal best; isolate the model from its preprocessing before crediting it. `09_XGBoost/05`. |
| any classifier (imbalanced) | "I'll report precision/recall at the threshold that maximises F1 on the test set." | That tunes a hyperparameter — the threshold — on the test set, so the reported precision/recall are optimistic. Choose the operating threshold on a **validation** slice, then read it **once** on test (the val-chosen threshold need not be test-optimal — that gap is the honesty). And lead with **PR-AUC**, not accuracy, when the positive class is rare. `09_XGBoost/05`. |
| XGBoost / LightGBM | "More estimators always help." | Without shrinkage/early stopping, more rounds overfit. |
| LightGBM | "Leaf-wise reaches a lower training loss than level-wise, so it is the better way to grow a tree." | That is a **training**-loss claim. Leaf-wise (best-first) spends each split where the gain is largest, so for a fixed leaf budget it fits the training data tighter — but greedily pouring splits down one branch is also how a tree **overfits**. Whether it generalizes is the `num_leaves` question (tuned in NB 4); leaf-wise's efficiency is its overfitting risk. (Built by hand, leaf-wise == LightGBM's single tree exactly.) `10_LightGBM/01`. |
| LightGBM | "Leaf-wise growth is a LightGBM invention." | It is the **modern default**, not LightGBM-only: scikit-learn's `HistGradientBoosting*` grows leaf-wise (`max_leaf_nodes=31`, ch 08 NB 5) and XGBoost offers `grow_policy='lossguide'` (ch 09 NB 4). LightGBM made it the default and built the library around it. `10_LightGBM/01`. |
| LightGBM | "num_leaves like max_depth." | Leaf-wise growth means large `num_leaves` overfits fast. |
| LightGBM / GOSS | "GOSS beats a random subsample because it keeps the rows that matter." | Only when the gradients are **concentrated**. GOSS's variance edge over a uniform subsample at the same fraction is gated by gradient concentration (top-20% share of Σ\|g\|), with a crossover near ~0.5. On dense tabular data concentration stays low (measured ~0.21→0.47 over 200 rounds), so GOSS only **ties** a uniform subsample there; its decisive win is large/wide/sparse, concentrated-gradient data (→ NB 5). `10_LightGBM/02`. |
| LightGBM / GOSS | "The `(1−a)/b` reweight is what makes GOSS good." | The reweight only keeps the sums **unbiased** (`H` exactly, `G` in expectation) — a uniform subsample with its `1/f` reweight is unbiased too. The lever is **variance**: GOSS keeps the dominant large-\|g\| rows exactly (zero sampling variance on them), so its gain estimate is tighter — but, again, only when a few rows dominate. `10_LightGBM/02`. |
| LightGBM / GOSS | "Switch on GOSS to train faster." | On dense moderate data the wall-clock is ~**flat** (measured): building feature histograms dominates, not the row count, so using fewer rows saves little time. GOSS's speed payoff appears when the row-pass dominates — very large `n` — the same regime where its gradients concentrate. `10_LightGBM/02`. |
| LightGBM / EFB | "EFB is just stacking one-hot columns together." | EFB bundles **approximately**-exclusive sparse features, tolerating a small **conflict rate** (rows where two bundled features both fire — Ke §4). That tolerance is what makes it more than concatenation, which forbids any overlap; it cuts histogram cost from `O(n·#features)` to `O(n·#bundles)`. `10_LightGBM/02`. |
| LightGBM / categorical | "A categorical feature has to be one-hot encoded before a tree can use it." | A gradient-boosted tree can split categories **natively**: summarise each category by its gradient statistic `G/H`, sort, and cut. No one-hot, and categories can be *grouped* (one-hot forbids that). `10_LightGBM/03`. |
| LightGBM / categorical | "Splitting K categories means trying all `2^(K−1)−1` groupings." | For a **convex** split criterion (the structure-score gain) the optimal binary partition is **contiguous** once categories are sorted by `G/H` — so only `K−1` candidates, not `2^(K−1)−1` (Fisher 1958). Exponential → linear, with no loss of optimality. `10_LightGBM/03`. |
| LightGBM / categorical | "Native categorical handling is always optimal and always best." | The exactness holds only for a convex criterion **and** a one-dimensional target (regression, or binary via `G/H` with `h=p(1−p)`); for **multiclass** it is a heuristic. And it is a convenience/speed win, not guaranteed higher accuracy — `min_data_per_group`/`cat_l2`/`cat_smooth` exist to stop a rare category overfitting. `10_LightGBM/03`. |
| LightGBM | "`num_leaves` and `max_depth` are the same capacity knob." | For leaf-wise growth `num_leaves` is **the** capacity dial; `max_depth` (default −1, unlimited) is an optional extra cap. Keep `num_leaves < 2^max_depth` (a leaf-wise tree with `2^d` leaves can run one branch to depth `2^d − 1`). A single tree overfits as `num_leaves` grows (peaks then falls); the boosted ensemble is far more robust. `10_LightGBM/04`. |
| LightGBM | "`min_child_samples` barely matters — leave it at 1." | It's the brake on leaf-wise's deepest branches (the min rows per leaf). On a single tree (or scarce data) it is decisive — too low memorizes, too high underfits, with a sweet spot (≈20). On an ample-data ensemble it is more a *safeguard* than a lever, but `=1` still invites memorizing leaves. `10_LightGBM/04`. |
| LightGBM | "Tuning should beat the defaults." | LightGBM's defaults are gentle (unlike XGBoost's aggressive eta0.3/depth6). On clean data they are already near the ceiling, so a `GridSearchCV` tune can land **within noise** of (even slightly below) the default on the sealed test. Tuning bites on larger/messier data; tune on train, read the test once. `10_LightGBM/04`. |
| LightGBM vs XGBoost | "Library X is faster / better than library Y." | On a fair benchmark the histogram boosters (LightGBM / XGBoost-hist / HistGBR) **tie on accuracy** (PR-AUC within ~0.001 at matched capacity). The speed winner is set by the **matching convention (tree shape)**: matched `num_leaves` → LightGBM fastest; library *defaults* → XGBoost faster (its depthwise-6 trees are smaller than LightGBM's leaf-wise 31) — that compares shapes, not engines. Only slowly does scale shift it. **No universal best; measure on your data.** `10_LightGBM/05`. |
| Data quality | "It has no `NaN`, so the data is clean." | A dataset can have **no NaN yet carry sentinel values** (MiniBooNE uses **−999** for undefined measurements). Gradient-boosted **trees handle a sentinel natively** (it becomes its own split — the ch 09 NB 3 idea, no imputation), but the extreme value **wrecks distance/scale-based methods** (PCA, scaled linear models) — set those rows aside or robust-scale for *those* steps only. `10_LightGBM/05`. |
| MLP / Neural Networks | "It won't converge." | Check scaling, learning rate, and seed before concluding anything. |
